### 📌 MVC 패턴

MVC는 소프트웨어 아키텍처 패턴 중 하나로, 애플리케이션을 **Model(모델), View(뷰), Controller(컨트롤러)** 3가지 역할로 관심사를 나누어 유지보수성과 확장성을 높이는 구조입니다.

- **Model (모델)**:
  - 애플리케이션의 **데이터 및 비즈니스 로직을 관리**하는 부분이다.
  - 서버에서 받은 데이터를 처리하거나 상태를 관리한다.
  - UI(뷰와 컨트롤러)에 대해서는 관여하지 않는다.
  - 변경되면 관찰자 객체에게 알려 뷰를 변경시킨다.
- **View (뷰)**:
  - UI 요소를 담당하며, 모델의 데이터를 시각적으로 표현한다.
  - 관찰자 패턴을 사용하여 모델이 변경되거나 수정될 때 변경된다.
  - React, Vue 같은 라이브러리가 이 역할을 한다.
- **템플릿(Template)**
  - 뷰를 생성하기 위해 사용 되는 방식으로 동적인 HTML 콘텐츠를 깔끔하고, 효율적이며, 유지보수가 용이한 방법으로 만들어 준다.
- **Controller (컨트롤러)**:
  - 사용자 입력을 받아 모델과 뷰를 연결하는 역할이다.
  - 이벤트를 처리하고, 모델의 상태를 업데이트하며, 변경된 데이터를 뷰에 반영한다.
  - 전통적인 MVC에서는 독립적인 역할이지만, 프론트엔드에서는 뷰와 합쳐지는 경우가 많다.

### 📌 MVP 패턴

- **Presenter(프레젠터)**
  - 뷰에 대한 UI 비지니스 로직을 담당하는 구성요소로, MVC와 달리 뷰에서의 이벤트 호출은 프레젠터로 위임된다.
  - 프레젠터는 뷰와 분리되어 있으며, 인터페이스를 통해 뷰와 통신한다. 이 방식은 단위테스트에서 뷰를 모킹할수 있는 등 많은 장점을 제공한다.

### 📌 MVVM 패턴

MVC와 MVP를 기반으로 하는 패턴으로, UI 개발 부분과 비지니스 로직 동작 부분을 선언적 데이터 바인딩을 활용하여 명확하게 분리하는 방식을 사용한다.
MVVM의 V는 능동적인 역할로, 데이터 바인딩, 이벤트, 동작등을 포함하고 있다.

- **View (뷰)**:
  - 사용자에게 **UI를 렌더링**하는 역할을 한다.
  - 사용자 인터페이스 요소들이 표시되는 곳으로, 화면의 구성 요소(버튼, 텍스트박스 등)와 관련된다.
  - View는 **사용자 입력을 처리**하고, **ViewModel에 알리거나 이벤트를 전달한다.**
  - **직접적인 데이터 로직**을 포함하지 않으며, **ViewModel**과 데이터 바인딩을 통해 데이터를 표시한다.
- **ViewModel (뷰모델)**:
  - **View**에 표시할 **데이터와 로직**을 제공한다.
  - ViewModel은 **Model**에서 데이터를 가져와서, **View**에 맞게 데이터를 가공하여 전달한다.
  - **View와의 양방향 데이터 바인딩**을 통해 **UI 상태 변화에 즉시 반응**할 수 있다.
  - **View**는 ViewModel의 **속성**을 데이터 바인딩을 통해 업데이트하고, **ViewModel**은 **View**의 변경 사항을 감지하고 반영한다.
  - **UI 로직**을 담당하며, **View의 상태를 관리**하고, **사용자 입력**을 처리하여 **Model**에 전달한다.
- **장점**
  - UI와 구동하게 해주는 요소를 동시 개발 가능하게 한다.
  - V를 추상화 함으로서 V뒤에 작성되는 비지니스 로직의 양이 줄어딘다.
  - VM은 이벤트 중심 코드에 비해 단위 테스트가 더 쉽다.
  - VM은 UI 자동화나 상호작용에 대한 고려 없이 테스트가 가능하다.
- **단점**
  - 단순한 UI의 경우 과한 구현이 될 수 있다.
  - 데이터 바인딩은 선언적이고 사용하기 편할 수 있지만, 당순히 중단점을 설명하는 명령형 코드에 비해 디버깅이 어려울 수 있다.
  - 복잡한 애플리케이션에서는, 데이터 바인딩이 상당히 관리 부담을 만들어 낼 수 있다. **양방향 데이터 바인딩**이 잘못 관리되면, 애플리케이션의 복잡도가 증가하면서 **성능**이나 **코드 유지보수**에 어려움을 겪을 수 있다.
  - **대규모 애플리케이션에서 MVVM 패턴을 적용할 때** **ViewModel**을 미리 설계하는 것이 **어려울 수 있다.**

### 📌 MV\* 패턴과 리액트

리액트는 선언형 프로그래밍 방식으로 애플리케이션이 원하는 상태를 기술하고, 그 상태를 기반으로 적절한 뷰를 렌더링한다. 리액트에서 MVC 패턴을 사용하지 않는 이유는 리액트에서는 서버가 브라우저에 뷰를 직접 제공하지 않고, 데이터를 제공하기 때문이다. 리액트는 이 데이터를 기반으로 브라우저에서 실제 뷰를 생성한다. 이런 관점에서는 리액트를 MVC 패턴의 뷰라고 할수는 있지만, 전통적인 의미에서의 MVC 프레임워크는 아니다.
